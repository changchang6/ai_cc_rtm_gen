# Testcase Patterns Reference

This document provides categorized testcase patterns for RTM generation. Testcases are **active, driving** components that generate stimuli and create test scenarios.

## Core Principle

**Testcase核心职责**: 生成激励，创造测试场景
**Testcase工作性质**: 主动的、驱动型的。它向设计施加"刺激"。
**Testcase关注焦点**: "做什么测试"。覆盖哪些功能点、场景、边界条件或协议要求。

---

## Testcase Standard Format

```
<testcase_name>
配置条件：
<配置步骤列表>
输入激励：
<激励步骤列表>
期望结果：
<期望行为和检查点列表>
coverage check点：
<覆盖率收集说明>
```

## Category 1: Configuration Testcases

### 寄存器配置测试

**使用场景**: 验证寄存器配置功能

**模板格式**:
```
test_<模块名>_reg
配置条件：
1.<复位配置>;
2.模块解复位，模块进入初始化工作状态
其它：<其他固定配置>
输入激励：
1.九步法寄存器测试激励;
期望结果:
1.寄存器读写等属性正常;
coverage check点 :
<覆盖率说明>
```

**示例** (来自RTM_CRG_BT):
```
test_crg_reg
配置条件:
1.crg_if配置模块复位信号（注意设计中寄存器模块的时钟来源为pll_gpu_clk_fr）;
2.模块解复位，模块进入初始化工作状态
其它：以上测试过程中，DFT接口控制信号按需求固接;
输入激励：
1.九步法寄存器测试激励;
期望结果:
1.寄存器读写等属性正常;
coverage check点 :
对总线接口以及地址和数据位宽收集功能覆盖率;
```

### 模式切换测试

**使用场景**: 验证工作模式切换

**模板格式**:
```
test_<模块名>_mode_switch
配置条件：
1.<初始配置>;
2.模块解复位，进入初始工作状态
输入激励：
1.配置模式A，验证功能正常；
2.切换到模式B，验证切换过程；
3.在模式B下验证功能正常；
期望结果：
1.模式切换无异常；
2.各模式下功能正确；
coverage check点：
对模式切换序列收集功能覆盖率
```

<期望行为列表>

```

**示例** (来自RTM_CRG_BT):
```
test_crg_normal

1. crg_if配置模块复位信号;
2.模块解复位，随机配置其他分频等配置信号，模块进入初始化工作状态;

### 时序违例测试

**使用场景**: 验证时序异常处理

**模板格式**:
```
test_<模块名>_timing_violation
配置条件：
1.<配置条件>;
输入激励：
1.通过随机化<信号>与时钟边沿的相对时间关系，触发时序违例；
2.验证同步逻辑的抗亚稳态能力；
期望结果：
模块无异常，时序违例被正确处理
```

**示例** (来自RTM_CRG_BT):
```
test_race_condition
配置条件：
1.配置分频比等控制信号;
输入激励：
1.通过随机化复位脉冲与时钟边沿的相对时间关系,使得复位信号在时钟边沿附近变化，触发时序违例;
2.强制异步信号直接输入同步器，验证双触发器同步链的抗亚稳态能力（时钟设计中已包含异步信号同步处理，因此只要check模块无异常即可）；
3、对时钟分频及时钟mux选择同时进行修改;
4、并行触发多个使能信号的翻转;
期望结果:
模块无异常
```

### 反压机制测试

**使用场景**: 验证时钟反压和级联关断

**模板格式**:
```
test_<模块名>_clk_back_pre
配置条件：
1.<配置条件>;
2.模块进入正常工作状态;
输入激励：
1.打开所有时钟门控；
2.依次关断下游时钟门控；
期望结果：
1.下游关断后，反压信号生成；
2.逐级向上反压关断；
coverage check点：
对反压序列收集功能覆盖率
```

**示例** (来自RTM_CRG_BT):
```
test_crg_clk_back_pre0
配置条件:
1. crg_if配置模块复位信号;
2.模块解复位，pre_div模块传入固定分频参数，模块进入初始化工作状态;
3.随机配置clk_core;
其它：以上测试过程中，DFT接口控制信号按需求固接;
输入激励：
1.打开所有输出前clk_gate;
2.时钟正常输出后，依次关断pll_ao_clk及pll_cpu_clk输出clk_gate;
期望结果:
1.clk_core模块先正常工作，关断pll_ao_clk clk_gate后，仅pll_ao_clk无输出;
2.继pll_cpu_clk的clk_gate关断后，两路的反压信号汇总，进而关断上一级gfree_mux组件;
3.clk_gate打开时钟路正常输出，关断时钟路无时钟输出。
coverage check点 :
对clk_core相关随机配置信号收集功能覆盖率;
```

## Common Mistakes to Avoid

1. **不要写成checker**: Testcase必须包含具体的配置步骤和输入激励
   - ❌ 错误: "检查信号X是否正确"
   - ✅ 正确: "配置寄存器Y，输入激励Z，检查信号X"

2. **必须包含四个部分**: 配置条件、输入激励、期望结果、coverage check点
   - 每个部分都必须明确具体

3. **引用具体名称**: 必须使用LRS中的具体信号名、寄存器名、opcode值
   - ❌ 错误: "配置使能信号"
   - ✅ 正确: "配置CTRL.EN=1"

4. **单一功能点**: 如果一个testcase测试多个功能点显得混乱，建议拆分
   - 每个testcase应聚焦一个主要功能点

5. **覆盖率说明**: 
   - 随机测试：说明收集哪些信号的功能覆盖率
   - 定向测试：说明"直接用例覆盖，不收功能覆盖率"
